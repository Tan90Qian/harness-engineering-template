# 测试策略指南

> 本文件定义项目的自动化测试策略。按项目实际情况修改覆盖范围和优先级。
> AI 生成测试代码时会参考此文件。

## 为什么 AI 工程化需要自动化测试？

AI 让代码生成速度提升数倍，但也意味着**引入缺陷的速度同步加快**。
自动化测试是唯一能以同等速度验证正确性的手段。

```
AI 生成代码 → 类型检查（结构对不对？）→ 单元测试（逻辑对不对？）
           → 集成测试（接口通不通？）→ E2E 测试（用户能不能用？）
```

## 测试金字塔

```
           ┌───────────────┐
           │   E2E 测试     │  ← 仅核心流程，不追求全覆盖
           │               │     成本高，维护重
           ├───────────────┤
           │  集成测试       │  ← API 接口测试
           │               │     覆盖 Controller 层
           ├───────────────┤
           │  单元测试       │  ← 工具函数 + 业务逻辑
           │               │     成本低，回报快
           └───────────────┘
```

## 各测试类型详解

### 1. 单元测试（Unit Test）

**定位**：验证最小代码单元（函数、类）的行为正确性。

| 维度 | 价值 |
|------|------|
| AI 护栏 | AI 生成的工具函数可立即通过单测验证，避免边界错误 |
| 重构安全网 | AI 重构代码时，单测确保行为不变 |
| 类型补充 | TypeScript 检查结构，单测验证运行时行为 |
| 活文档 | AI 读取单测后能理解函数预期行为，生成代码更准确 |

**适用范围**：
- 工具函数（必测）
- 状态机流转逻辑
- 数据格式化/转换
- 业务规则校验

**推荐工具**：Vitest / Jest

**投入产出比**：⭐⭐⭐⭐⭐

### 2. 集成测试（Integration Test）

**定位**：验证多个模块协作时的行为正确性，尤其是接口层。

| 维度 | 价值 |
|------|------|
| 接口保障 | 确保 Controller → Service → 响应的完整链路正确 |
| 契约验证 | 共享类型定义契约，集成测试验证实际返回是否符合契约 |
| Mock 切换 | 初期 Mock 开发，集成测试确保切换真实后端时行为一致 |

**适用范围**：
- API Controller 接口（必测）
- 数据库/缓存交互层
- 第三方服务调用（Mock 外部，测内部逻辑）

**推荐工具**：Supertest（Node.js）/ Testing Library

**投入产出比**：⭐⭐⭐⭐

### 3. 端到端测试（E2E Test）

**定位**：模拟真实用户操作，验证整条链路的正确性。

| 维度 | 价值 |
|------|------|
| 用户视角 | 唯一从用户角度验证的手段 |
| 跨层回归 | API 返回结构变了，前端能否正常渲染？ |
| 视觉回归 | 截图对比可发现 AI 生成的 UI 样式偏差 |

**适用范围**：
- Web 核心业务流程（3-5 条即可）
- 登录 → 浏览 → 操作 等关键路径

**推荐工具**：Playwright / Cypress

**投入产出比**：⭐⭐⭐（成本较高，仅覆盖关键路径）

### 4. 变异测试（Mutation Test）

**定位**：评估现有测试的**质量**——自动修改源代码，检验测试是否能捕获变化。

| 维度 | 价值 |
|------|------|
| 测试质量度量 | 覆盖率 ≠ 测试质量，变异测试揭示真实有效性 |
| AI 测试验证 | AI 常生成"断言薄弱"的测试，变异测试能暴露空洞测试 |
| 关键逻辑加固 | 对金额计算、状态流转等核心逻辑确保测试真的在保护 |

**适用范围**：
- 核心工具函数的测试质量验证
- 金额/佣金计算逻辑
- 状态机流转规则

**推荐工具**：Stryker

**投入产出比**：⭐⭐（运行耗时长，建议仅对关键模块定期运行）

## 优先级建议

| 优先级 | 测试类型 | 覆盖目标 | 引入时机 |
|--------|---------|---------|---------|
| P0 | 单元测试 | 工具函数 ≥90%，业务逻辑 ≥80% | 项目启动即引入 |
| P1 | 集成测试 | API Controller 100% | 第 2 周 |
| P2 | E2E 测试 | 3-5 条核心流程 | 功能基本完成后 |
| P3 | 变异测试 | 核心模块定期运行 | 稳定期按需 |

## CI 集成

```yaml
# 在 CI 流水线中按顺序执行
- name: 单元测试
  run: pnpm test

- name: 集成测试
  run: pnpm test:integration

- name: E2E 测试
  run: pnpm test:e2e
  if: github.event_name == 'pull_request'
```

## AI 与测试的关系

```
AI 生成代码
    │
    ▼
┌─────────────────────────────────────┐
│  tsc --noEmit    → 类型对不对？      │  ← 静态分析
│  ESLint          → 风格和潜在错误？   │
├─────────────────────────────────────┤
│  单元测试        → 逻辑对不对？       │  ← 最快反馈
│  集成测试        → 接口通不通？       │  ← 接口保障
├─────────────────────────────────────┤
│  E2E 测试        → 用户能不能用？     │  ← 最终验证
├─────────────────────────────────────┤
│  变异测试        → 测试本身靠谱吗？   │  ← 元验证
└─────────────────────────────────────┘
```

**核心原则**：测试不是目的，**可信赖的快速反馈**才是目的。
在 AI 加速开发的场景下，自动化测试的价值是"以机器速度验证机器（AI）产出的正确性"。
